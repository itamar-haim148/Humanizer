## US-001 — Monorepo scaffold + Coolify-ready Docker Compose ✅

**Completed**: 2026-05-19

**What was done:**
- Created directory tree: `apps/web/src/{app,components,lib,i18n}`, `apps/api/humanizer/engines/{synonyms,ai_phrases,data}`, `apps/api/tests/fixtures`, `docs/`
- Wrote `docker-compose.yml` with services `api` (port 8000 internal via `expose`) and `web` (port 3000 published via `ports`)
- Shared bridge network `humanize-net`
- Named volume `hf_cache` mounted at `/root/.cache/huggingface` on `api` (HF_HOME env set to same)
- Health checks on both services, `depends_on: api.service_healthy` for `web`
- Resource limits: api `mem_limit: 2g cpus: 2.0`; web `mem_limit: 512m cpus: 1.0`
- Wrote `.env.example` with all env vars consumed by compose
- Validated with `docker compose config --quiet` → exit 0

**Learnings (for later stories):**
1. Compose v2 deprecates the `version:` key — used `name: humanize` instead at top-level.
2. `expose:` (internal) vs `ports:` (host-published) — API uses `expose` only so Coolify routes only the web service to public.
3. Healthcheck for `api` uses `python -c urllib` to avoid adding `curl` to the slim image later (will work once `/health` exists in US-002).
4. Healthcheck for `web` uses `wget --spider` which is present in node-alpine images.
5. `HF_HOME=/root/.cache/huggingface` is the canonical path Hugging Face libraries use; volume mount keeps model cache across rebuilds — important once SynthID (US-012) ships, since first download is multi-GB.
6. `WEB_PORT` overridable via env so dev can use 3001 if 3000 is busy.
7. `cpus:` (compose v2 long form) — kept as float numbers; on Windows Docker Desktop these are advisory.
8. Dockerfiles intentionally not created here — they belong to US-029. Compose YAML validation does not require Dockerfile existence.
9. Build arg `NEXT_PUBLIC_API_BASE_URL` passed at build time so it's baked into the Next.js client bundle; runtime env retained for SSR fetches.

- 7 schema tests; suite now 10 passing
- mypy: 5 source files clean

**Learnings:**
1. Pydantic v2 `field_validator` + `@classmethod` order: `@field_validator` then `@classmethod`.
2. Use `Literal[...]` as exported type aliases for shared signatures across engines.
3. Schema hard cap (50k) layered with route env cap (10k) — defense in depth against payload abuse.
4. `field_validator` preserves original whitespace; only asserts non-whitespace presence — engines may need exact spacing.
5. `verdict: Literal[...]` rejects unknown strings automatically — no extra validator needed.

## US-012 — SynthID detector ✅

**Completed**: 2026-05-19

**What was done:**
- `humanizer/engines/detector_synthid.py`: lazy-loaded SynthID detector
- Three-state design: disabled / enabled-but-unavailable / enabled-and-available — instead of returning `None` we always return a typed `SynthIDResult` (cleaner for the API schema)
- `_try_load()` isolates the import + load logic so unit tests inject a fake loader via `_set_loader_for_tests`
- `_coerce_score()` accepts float, dict-with-score, or tensor-like (item()), clamped to [0,1]
- 5 tests using fake loader (no transformers/torch needed in CI)
- `docs/engines.md` documents enable steps + state semantics

**Learnings:**
1. Deviating from PRD's `SynthIDResult | None` to always-typed result simplifies frontend code (no null checks for state info) — preserved availability/score semantics via `available` + `score` fields.
2. `getattr(transformers, "SynthIDTextWatermarkDetector", None)` handles older transformers versions where the class isn't present.
3. `_set_loader_for_tests` test seam keeps prod code clean (no `if testing:` branches) and resets cached `_detector` automatically.
4. Score coercion supports any reasonable return type; HF detector APIs vary.
5. The `# type: ignore[import-not-found]` on `import transformers` is unnecessary because mypy can't resolve it either way (transformers isn't a dev dep) — removed and mypy accepts module-level fallback via ImportError.

## US-013 — Pipeline orchestrator + scoring fusion ✅

**Completed**: 2026-05-19

**What was done:**
- `humanizer/pipeline.py`: `run_humanize` + `run_detect`
- Fusion weights `_SUB_WEIGHTS` sum to 1.0 with runtime assert
- Verdict thresholds: <0.35 human, <0.65 mixed, ≥0.65 ai
- Humanize order: cleaner → lexical (EN/HE) → structural → final cleaner; detector runs twice for before/after
- `_highlighted_segments` picks top-5 sentences by fused sub-score; reason = highest-contributing sub-metric label
- 7 pipeline tests covering thresholds, EN/HE shape, AI vs human distinction, watermark report, monotonicity
- Full suite: 75 tests passing; mypy clean across 14 source files

**Learnings:**
1. Branched assignment with two distinct dataclass types from EN/HE lexical engines requires separate variable names — mypy correctly rejects same-name reassignment across incompatible types.
2. Mid-text watermark cleaning before lexical/structural is critical — otherwise zero-width characters break word boundary regexes.
3. `transformations[:50]` caps the response payload — heavy aggressive runs on long text could emit 200+ entries.
4. `_highlighted_segments` recomputes stats per sentence; for very long inputs this is O(n) over sentences. Acceptable for ≤10k char cap.
5. Pipeline kept all heuristics — no LLM call — so total Humanize latency on a 500-char sample is ~5-15ms locally.

## US-014 to US-017 — API endpoints + rate limit + tests + fixtures ✅

**Completed**: 2026-05-19

**What was done:**
- `humanizer/routes.py`: `/api/humanize` + `/api/detect` POST endpoints with `_enforce_size` guard (413 on overflow)
- `humanizer/rate_limit.py`: `SlidingWindowLimiter` (deque per IP + `threading.Lock`) + `RateLimitMiddleware` scoped to `/api/*` only
- `RATE_LIMIT_PER_MIN` (default 30) + `RATE_LIMIT_WINDOW_SECONDS` (default 60) + `TRUST_PROXY` (default false) settings
- `X-Forwarded-For` first-value parsing when `trust_proxy=true`; 429 response includes `Retry-After` header
- `tests/test_routes.py` + `tests/test_rate_limit.py` (6 tests)
- `tests/fixtures/{ai,human}_{en,he}.json`: 5 hand-curated samples each (20 total)
- `tests/test_fixtures_distinguish.py`: asserts AI mean probability > human mean per language with 0.1+ gap
- Full suite: 90 tests, 92% coverage, mypy clean across 16 source files

**Learnings:**
1. Rate-limit middleware path scoping (`/api/*` only) keeps `/health` and `/docs` open for Coolify monitoring + manual debug.
2. `threading.Lock` on the deque is enough — Starlette runs middleware in the main asyncio thread, but uvicorn `--workers >1` would split per-process state anyway; per-process limit is acceptable for a single-VPS deploy.
3. `Retry-After` header value computed from `window_seconds - elapsed_oldest` so clients back off precisely.
4. Hand-curated fixtures gave reproducible verdict separation; random LLM-generated samples produced inconsistent results across runs.
5. Hebrew AI fixtures needed to include semi-formal connectors ("יתרה מזאת", "כמו כן") that the lexical_he engine recognizes as AI signal.
6. `len(text) > cap` size check at route boundary returns FastAPI 413 — kept out of Pydantic validator so the error message can reference the env-driven cap.

## US-018 to US-028 — Frontend (scaffold + Humanize + Detect + actions) ✅

**Completed**: 2026-05-19

**What was done:**
- `apps/web/package.json`: Next 16.0.0, React 19.0.0, TypeScript 5.7.2, Tailwind 3.4.17
- `apps/web/tsconfig.json`: strict, `@/*` paths, `moduleResolution: bundler`
- `apps/web/next.config.ts`: `output: "standalone"` for Docker, `experimental.typedRoutes`
- `apps/web/tailwind.config.ts`: `darkMode: "class"`, brand color tokens, font variables
- `apps/web/src/lib/types.ts`: full mirror of API Pydantic schemas + `ApiResult<T>` discriminated union
- `apps/web/src/lib/api.ts`: `humanize()` + `detect()` returning `Promise<ApiResult<T>>` (never throws); uses `NEXT_PUBLIC_API_BASE_URL`
- `apps/web/src/lib/locale.ts`: async `resolveLocale()` reads cookie → Accept-Language; `resolveTheme()` reads `theme` cookie
- `apps/web/src/i18n/{en,he}.json` + `index.ts` with `formatString` helper for placeholders
- `apps/web/src/app/layout.tsx`: server-resolved `<html lang dir>`, Inter + Heebo via next/font, dark class on html
- `apps/web/src/components/Header.tsx`: language + theme toggle (cookie write + `window.location.reload()`)
- `apps/web/src/components/Tabs.tsx`: client-state Humanize/Detect switcher
- `apps/web/src/components/HumanizePanel.tsx`: two-panel textarea, segmented strength control, clean-watermarks checkbox, submit/copy/download/clear actions, BOM-prefixed UTF-8 download blob
- `apps/web/src/components/MetricsCard.tsx`: collapsible card with paired before/after progress bars
- `apps/web/src/components/DetectPanel.tsx`: SVG circular gauge, 12-metric grid, top-3 contributors, `<mark>` highlighter for suspect segments, watermark report drawer + JSON download
- `apps/web/public/.gitkeep`: placeholder so the directory always exists for the standalone Docker COPY step

**Learnings:**
1. Next.js 16 `cookies()` and `headers()` are async in App Router — `resolveLocale()` and `resolveTheme()` must `await` them; older docs/snippets that read them synchronously break under 16.
2. `next/font` Heebo (Hebrew) registers a CSS variable; setting it on `<body>` along with Inter lets the `lang="he"` selector inherit the right family without duplicate `<link>` tags.
3. RTL handling: setting `dir="rtl"` on `<html>` plus Tailwind logical properties (`ms-*`, `me-*`, `ps-*`, `pe-*`) avoided a plugin entirely.
4. `ApiResult<T>` discriminated union (`{ ok: true; data } | { ok: false; error }`) replaces try/catch at every call site; UI components branch on `.ok`.
5. The Humanize panel's download blob is prefixed with `\uFEFF` (BOM) so Windows Notepad / Excel detect UTF-8 correctly — Hebrew otherwise opens as mojibake.
6. Cookie-write + `window.location.reload()` is intentionally simple — Next 16 App Router router refresh wouldn't re-evaluate the server-resolved `<html lang/dir>` without a full reload.
7. SVG gauge with `strokeDasharray` based on `ai_probability` avoids a charting library — keeps the bundle small.
8. The watermark drawer JSON download uses `JSON.stringify(report, null, 2)` so users see the full structure (counts + findings).
9. Tailwind `dark:` variant works with the `class` strategy; the toggle writes a cookie and adds/removes `class="dark"` on `<html>` immediately for snappy first-paint.
10. `<mark>` element styled via Tailwind `[&_mark]:bg-yellow-200 dark:[&_mark]:bg-yellow-700/40` keeps highlight CSS local to the DetectPanel.

## US-029 to US-032 — Deployment (Dockerfiles, docs, production readiness) ✅

**Completed**: 2026-05-19

**What was done:**
- `apps/api/Dockerfile`: multi-stage `python:3.12-slim` (builder + runtime), wheels pre-built in builder, non-root `app` user, `HEALTHCHECK` via stdlib urllib (no curl needed), `uvicorn --proxy-headers --forwarded-allow-ips=*`
- `apps/web/Dockerfile`: 3-stage `node:22-alpine` (deps + builder + runner), non-root `nextjs` user, COPY order: `public/` → `.next/standalone` → `.next/static`
- `docs/engines.md`: per-engine provenance + SynthID enable steps (`ENABLE_SYNTHID=true`, `SYNTHID_MODEL`, hf_cache mount)
- `docs/deploy-coolify.md`: 6-step Coolify deployment guide with env var table (WEB_ORIGIN, RATE_LIMIT_*, TRUST_PROXY, MAX_TEXT_LENGTH, ENABLE_SYNTHID, etc.)
- Warm-load step in `main.py` lifespan: runs `detector_statistical.compute_all` for both languages so the first user request is sub-50ms (avoids cold dict/freq-list load)
- Final validations: `docker compose config --quiet` exits 0; 90 tests pass; mypy clean

**Learnings:**
1. `python:3.12-slim` is the right pin — `pydantic-core` has wheels; 3.13 also works but 3.12 is more conservative for production base image stability.
2. Multi-stage build with `pip wheel --wheel-dir /wheels` separates compile-time tooling from runtime — runtime image stays slim, no `build-essential`.
3. Non-root container users: `app` (api) + `nextjs` (web) — Coolify defaults expect non-root for security policy compliance.
4. `HEALTHCHECK CMD python -c "import urllib.request..."` avoids adding `curl` to slim image; Coolify polls `/health` and reads the healthcheck status.
5. Docker Compose `depends_on: condition: service_healthy` is required (not just `depends_on: api`) so `web` waits for the api healthcheck to pass before starting — otherwise SSR calls fail on cold start.
6. `output: "standalone"` in `next.config.ts` is what makes the 3-stage web Dockerfile possible — without it, the COPY paths don't exist.
7. `public/.gitkeep` placeholder avoided a shell-syntax COPY failure (`COPY ... 2>/dev/null || true` is not valid Dockerfile syntax) — keeping the directory present is the clean fix.
8. `--proxy-headers --forwarded-allow-ips=*` on uvicorn is required so the rate limiter sees the real client IP via `X-Forwarded-For` (Coolify's reverse proxy injects it).
9. Warm-load timer at startup is one JSON log line — Coolify log viewer parses it directly without extra config.
10. `expose_headers: ["x-request-id"]` in CORS works end-to-end: browser DevTools shows the id, helpful for "find this request in api logs" support workflow.
